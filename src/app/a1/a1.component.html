A1: Widget Factory in Adobe Spectrum

In this assignment, you are working for My Widget Factory, a new disruptor in the widget-making industry. They have recently designed their line of custom widgets, and want your help getting their online store to market!

Luckily for you, My Widget Factory's lead developer spent a bit of time making a basic store. The store allows a person to select, customize, and order widgets. But unfortunately, it looks like it was designed in the late-90's, and they are worried that this will impact sales. So, they want you to modernize the look and feel with a Design System.

This assignment has three parts, with a part due each week. In the first week (Part 1), you will first mock up a design for your online store. You will then implement your design using Spectrum's Web Components, with a check-in midway (Part 2) and a final implementation (Part 3). This assignment is worth 25 points, distributed among the three parts.

A Tour of the Factory

There are three pages that you will need to support: an order page, a shipping page, and a summary page. The images below show examples from the functional implemetation using plain HTML and a tiny bit of CSS, provided to you in the default_html folder.

Order Page

The factory supports the sale of four types of widgets: a Toy Boat, a Paper Plane, a Stuffed Animal, and a Train Car. To keep shipping costs down, the company wants to limit online orders to a max of 10 of each widget. The factory is particularly excited that all widgets are made-to-order. This means that each widget can be personalized to be whatever color of the customer decides!

After years of in-person sales, the company has set a price point for each widget where they can still turn a profit, and gathered an average "star rating" out of five from their customers. The Train Cars are particularly expensive to produce, and not very well-liked by customers... maybe they will phase them out soon.

Shipping Page

For the moment, the company is only able to ship widgets within the U.S., which simplifies the order fields. They support three shipping speeds: "express", "standard", and "slow", which each have different price points. Once shipping is specified, the customer can place an order!

Summary Page

Finally, each order gets returned with an invoice summarizing what was purchased. The company is somewhat flexible with what shows up in the invoice. They know that customers want to see the total price, shipping destination, and arrival date. But other information is optional or can be reformatted.

Overall, My Widget Factory is open to improvements to the design of their store. Take their current website as an example, but feel free to be creative or think more critically about the potential needs of their customers.

Starter Code

The code is available for download on GitHub. If you want to practice version control, you can use create your own copy of the repository via "use this template" in the top-right. You can also directly clone/download the repository via the "code" button.

Repository Structure

We have provided eight files (and one folder) to you. You will edit some of them in this assignment, but not all.

The files you should not need to edit are:

  • .gitignore, which specifies what files should not be a part of version control.
  • favicon.ico, which specifies the icon to render in your browser bar to represent the online store. You can choose to change this if you like, such as to make consistent theming in your design. But doing so is not required.
  • shipping_data.js and widgets_data.js. These two files include data about the respective prices and ratings for the widgets, and shipping costs and duration. This data is referenced in script.js, but you may also find it useful to look at.
  • script.js. This file includes all of the coding logic for storing information about desired widgets and shipping costs between pages, and calculating the total cost for the summary page.
  • You may additionally find it useful to refer to the default_html folder, which provides a functional implementation of the store.

Note that this means you should not edit any JavaScript files. You may choose to, but only for adding functionality beyond the assignment requirements.

The files you should edit are:

  • index.html: The HTML file for the order page for your store, where a customer selects what widgets to order and of what color.
  • shipping.html: The HTML file for the shipping page, where a customer selects where to send the widgets and what speed of shipping.
  • summary.html: The HTML file for the order summary page, where a customer can see their order summarized.
  • style.css: The CSS file to style all three pages on your site. It is possible to complete the assignment without editing this file. But you may find it useful for using custom CSS in your design.
  • readme.docx: A document with some questions to fill out about your experience using a Design System.
  • You will also need to add a file (either mockup.fig or mockup.xd) for your design mockup. See Part 1.

Editing and Running your Code

We recommend Visual Studio Code as an Integrated Development Environment (IDE). But you can use whatever program you want to edit your code. To run your site, you can open it up in your favorite browser.

On the order page and the shipping page, the coding logic (script.js) is expecting specific HTML ids to be used to indicate input fields. On the order page (index.html), these inputs are of the form *_quantity and *_color, where * refers to each of the four types of widgets (toyboat, paperplane, stuffedanimal, and traincar). The shipping page (shipping.html) expects ids for the address, city, state, and zip, as well as the shipping_speed. You might notice that the company's servers take a bit of time (about two seconds) to process shipping information and prepare the order. If you have an HTML element that you want to show up while waiting, you can give it the id of loading_progress, plus the hidden attribute.

On the summary page (summary.html), the coding logic (script.js) uses specific HTML ids to then re-populate information the customer entered about their order and summarize it. The page uses the same ids for *_quantity and *_color to refer to the amount and color of each widget ordered. If you want to summarize the total cost of shipping a type of widget, the coding logic also populates ids for *_subtotal. The coding logic similarly uses the same ids for address, city, state, zip, and shipping_speed, plus a shipping_subtotal to summarize the cost of shipping. Finally, the id of arrival_date is used to indicate when the package will arrive.

We expect that it will be useful to refer to the default_html folder to understand how these different ids are used in practice.

Spectrum requires that all components are nested inside of an sp-theme tag, which can be customized to be light, dark, etc. We have provided one for you in each of the .html files, along with a TODO comment indicating where you should placing your code.

For the first part of the assignment, mock up a design for the online store. You must make your mockup following an existing Design System, as you will later use components of that system in your implementation.

After consulting with the IN4MATX 285 course staff, My Widget Factory has specifically asked you to use Adobe's Spectrum, a powerful Design System that has an implementation of Web Components. There is a widely-used UI Kit for Spectrum on Figma. Spectrum also maintains a Plugin for Adobe XD. You are welcome to use either, though Figma is recommended since Adobe is stopping support for XD.

We will focus on Spectrum Components in this course. Adobe's Design Sytem has a wide range of different suggestions for how to implement an interface, from recommendations around voice and tone to words to include or exclude in your product. These are all excellent resources to look over, and offer useful advice on interaction design principles. We will primarily focus on the implementation aspects of the system in this assignment.

You can design your store however you wish, within a few constraints. You must:

  • Support all functionality demonstrated in the basic web store provided to you. This includes the sale of the four widgets, and displaying price and rating information. A customer can select how many to purchase (up to 10 of each). You must support entering shipping information, and summary page with the total cost and arrival date. It's up to you whether to include subtotals, unit prices, etc. in the summary.
  • Implement your store as three separate pages, one for the order, one for the shipping information, and one for a summary. The lead developer has implemented the backend of web store assuming this separation.
  • Implement the store to be functional on large screens (e.g., traditional Desktop browsers). Although My Widget Factory feels that it would be ideal for their store to look good on a mobile browser, they recognize that you are limited in your time.

Exactly how you go about supporting these functionalities is up to you. Spectrum has a long list of components in their Design System which can assist in creating your design. Overall design principles (e.g., interaction design) are beyond the scope of this class, but hopefully your prior courses have offered some guidance on how to approach this task.

You may choose to customize any of the visual components of the store. For example, we have provided images licensed under Creative Commons (with attribution) as examples of the four widgets. You are welcome to find new images or icons, change how the attributions are presented, or not include images at all. You can define a theme, choose typography and iconography, and implement other concepts like motion as you see fit.

Overall, we recommend creating your design using only the components of the Design System, as this will make implementation in the later stages of the assignment much easier. You're certainly welcome to introduce components not supported by the Design System. You may find later on that you'll have to modify your design, especially if you're new to HTML and CSS.

We recognize that this may be a short deadline to complete a full mockup, but we do ask that you complete your mockup by the assigned due date. The main objective of this assignment is learning to implement a design, and experience firsthand technical barriers which may arise when trying to do so. Do the best you can within the time constraints. Feel free to cut corners with your mockup. For example, the mockup does not need to be interactive, you can set a fixed arbitrary resolution. Overall, aim for a mockup that meets the above specifications, but we recommend not spending a lot of time optimizing your design to create an ideal user experience.
